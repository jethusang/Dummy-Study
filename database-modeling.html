Introduction to Database Modeling

In the late 1960s, mathematician Dr. E.F. Codd at IBM suggested using mathematical principles for database design. His seminal 1970 paper introduced the relational model, forming the foundation for relational databases and the normalization methodology.

Normalization addresses problems like data duplication and anomalies during insertion, deletion, and updates. This methodology determines database contents independently of physical constraints.

Key Insight: Normalization creates relational schemas with no information redundancy, eliminating data anomalies and ensuring data integrity.

Advantages of Databases

  • Reduced data redundancy
  • Reduced updating errors and increased consistency
  • Greater data integrity and independence
  • Improved data access through query languages
  • Reduced costs for data entry, storage, and retrieval

Database Implementation Challenges

  • Complex, difficult, and time-consuming design process
  • Substantial hardware and software start-up costs
  • Database damage affects all applications
  • Extensive conversion costs from file-based systems
  • Initial training required for programmers and users

Entity

Click to flip

An entity is any object in the system that we want to model and store information about. Entities are usually recognizable concepts, either concrete or abstract, such as person, places, things, or events which have relevance to the database.
Example: Employee, Student, Lecturer

Key Concepts

  • Entity

    Object we want to model and store information about

  • Attribute

    Item of information stored about an entity

  • Relationship

    Association between entities

  • Primary Key

    Uniquely identifies each record in a table

Relationship Types

One-to-One (1:1)

One occurrence of an entity relates to only one occurrence in another entity

One-to-Many (1:M)

One occurrence in an entity relates to many occurrences in another entity

Many-to-Many (M:N)

Many occurrences in an entity relate to many occurrences in another entity

Key Types

Primary Key

Uniquely identifies each record

Foreign Key

Links to a primary key in another table

Composite Key

Combination of two or more columns as a key

Secondary Key

Candidate keys not chosen as primary key

Database Normalization Process

Eliminates anomalies

Normalization Steps

1

UNF to 1NF

Convert unnormalized table to first normal form by removing repeating attributes to a new table.

2

1NF to 2NF

Remove partial dependencies by moving attributes that depend on only part of the key to a new table.

3

2NF to 3NF

Remove transitive dependencies by moving attributes that depend on other non-key attributes to a new table.

Normalization Rules Summary

First Normal Form (1NF)

  • Contains no repeating columns
  • Each cell contains a single value
  • Each record is unique

Second Normal Form (2NF)

  • Already in 1NF
  • Contains only columns dependent on the whole primary key
  • No partial dependencies

Third Normal Form (3NF)

  • Already in 2NF
  • All non-key columns are dependent only on the primary key
  • No transitive dependencies

Anomalies Addressed

  • Update anomalies
  • Delete anomalies
  • Insert anomalies
