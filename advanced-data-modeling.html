Advanced Data Modeling Concepts

The Extended Entity Relationship (EER) model enhances the original ER model with additional semantic constructs, including entity supertypes and subtypes, specialization hierarchies, and entity clusters.

These advanced concepts allow for more accurate representation of real-world data relationships and constraints, particularly in complex database systems.

Key Insight: Specialization hierarchies enable attribute inheritance, where subtypes inherit attributes and relationships from their supertypes, reducing redundancy and improving data integrity.

Entity Clusters

  • "Virtual" entity type representing multiple entities and relationships
  • Abstract - not an actual entity in the final ERD
  • Used to simplify complex ER diagrams
  • Avoids display of attributes when clusters are used

Key Concepts

Subtype Discriminator

Attribute that determines which subtype a supertype occurrence belongs to

Attribute Inheritance

Subtypes inherit all attributes and relationships of their supertype

1:1 Relationship

At implementation, supertype and subtype maintain 1:1 relationship

Specialization Constraints

Disjoint

Subtypes contain unique subsets of supertype

EMPLOYEE subtypes:
• PILOT
• MECHANIC
• ACCOUNTANT

Overlapping

Subtypes contain nonunique subsets

PERSON subtypes:
• EMPLOYEE
• STUDENT
• ADMINISTRATOR

Partial

Supertype occurrence doesn't have to belong to a subtype

Total

Every supertype occurrence must belong to at least one subtype

Specialization vs Generalization

Specialization

Top-down process of identifying more specific subtypes from a generic supertype

  • Start with general entity (e.g., EMPLOYEE)
  • Identify specialized roles (e.g., PILOT, MECHANIC)
  • Group unique characteristics and relationships

Generalization

Bottom-up process of identifying a generic supertype from specific subtypes

  • Start with specific entities (e.g., PILOT, MECHANIC)
  • Identify common characteristics
  • Create generalized supertype (e.g., EMPLOYEE)

Primary Key Strategies

Primary Key Guidelines

Unique Values

Must uniquely identify each entity instance

Nonintelligent

Should not have embedded semantic meaning

No Change Over Time

Should remain constant once assigned

Preferably Single-Attribute

Simplifies implementation of foreign keys

Security Compliant

Must not be composed of sensitive attributes

Key Types

Natural Key

Real-world identifier (e.g., SSN, ISBN)

Pros:
• Real-world meaning
• Familiar to users
Cons:
• May change over time
• Security concerns

Surrogate Key

System-generated identifier (e.g., auto-increment ID)

Pros:
• Never changes
• No business meaning
Cons:
• No real-world meaning
• Requires join for business context

Composite Key

Combination of two or more attributes

Use Cases:
• M:N relationships (junction tables)
• Weak entities

When to Use Composite Keys

M:N Relationships

In junction tables to ensure unique combinations

Example:

ENROLL(CLASS_CODE, STU_NUM) ensures a student can't enroll in same class twice

Weak Entities

For entities dependent on another entity

Example:

DEPENDENT(EMP_NUM, DEP_NUM) - dependents exist only in context of employee

Data Modeling Design Cases

Case #1: 1:1 Relationships

Implementing one-to-one relationships with proper foreign key placement.

Guidelines:
  • One side mandatory, other optional: Place PK of mandatory side in optional entity
  • Both sides optional: Place FK in entity that causes fewest nulls
  • Both sides mandatory: Consider combining entities

Case #2: Time-Variant Data

Maintaining history of data that changes over time.

Solution:
  • Create separate history tables
  • Include effective date and end date
  • Use surrogate keys for history records

Case #3: Fan Traps

Occurs when one entity is in two 1:M relationships to other entities.

Problem:

Produces an association among other entities not expressed in model

Solution:

Properly identify relationships to eliminate the trap

Case #4: Redundant Relationships

Multiple relationship paths between related entities.

Problem:

Can lead to inconsistency and maintenance issues

Solution:

Remove unnecessary relationships to simplify the model

Data Modeling Checklist

Model Verification

  • All entities and relationships properly defined
  • All attributes assigned to appropriate entities
  • All primary keys properly specified
  • All foreign keys properly specified

Design Validation

  • Constraints properly identified
  • Redundant relationships removed
  • Time-variant data handled appropriately
  • Fan traps eliminated
